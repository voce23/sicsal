1 ¿Qué son las migraciones?

Las migraciones son archivos PHP que definen la estructura de tus tablas en la base de datos. Son como un "control de versiones para la BD".

SQL directo Migraciones Laravel
Escribes CREATE TABLE … en MySQL Escribes PHP: $table->string('nombre')
Si cambias algo, tienes que recordar el ALTER Laravel registra qué migró y qué falta
Cada desarrollador ejecuta SQL manualmente Todos ejecutan php artisan migrate
No hay forma fácil de revertir php artisan migrate:rollback deshace
ℹ️
Plan de la Base de Datos
SICSAL tiene 50 migraciones que crean ~35 tablas. Las crearemos en 6 lecciones (06-11). Empezamos con las más simples: municipios y centros_salud.

2 Crear una migración con Artisan

Artisan genera el esqueleto de la migración:

Terminal bash 
$ php artisan make:migration create_municipios_table
INFO  Migration [database/migrations/2026_04_03_015720_create_municipios_table.php] created successfully.

El nombre del archivo tiene formato: YYYY_MM_DD_HHMMSS_nombre.php. La fecha asegura que las migraciones se ejecutan en orden cronológico.

💡
Convención de nombres
Usa create_XXX_table para tablas nuevas y add_YYY_to_XXX_table para agregar columnas a tablas existentes. Laravel usa el nombre para inferir la operación.

El archivo generado contiene dos métodos:

Esqueleto de migración (generado por artisan) php 
return new class extends Migration
{
    public function up(): void
    {
        // Aquí defines la estructura de la tabla
    }

    public function down(): void
    {
        // Aquí la reviertes (drop, eliminar columnas, etc.)
    }
};
  • up() — Se ejecuta cuando corres php artisan migrate
  • down() — Se ejecuta cuando corres php artisan migrate:rollback

3 Tabla municipios

La tabla municipios almacena los municipios del departamento. Es la tabla más simple y no depende de ninguna otra.

Diseño de la tabla

Columna Tipo Descripción
PK id bigint Identificador autoincremental
nombre varchar(100) Nombre del municipio (ej: "Capinota")
departamento varchar(100) Departamento (default: "Cochabamba")
activo boolean Borrado lógico (true/false)
created_at timestamp Fecha de creación (automática)
updated_at timestamp Fecha de última modificación (automática)

Código completo de la migración

Abre el archivo generado en database/migrations/ y reemplaza todo el contenido:

database/migrations/XXXX_XX_XX_XXXXXX_create_municipios_table.php php 
<?php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    public function up(): void
    {
        Schema::create('municipios', function (Blueprint $table) {
            $table->id();                                      // PK autoincremental
            $table->string('nombre', 100);                      // varchar(100)
            $table->string('departamento', 100)
                  ->default('Cochabamba');                    // default
            $table->boolean('activo')->default(true);         // borrado lógico
            $table->timestamps();                               // created_at + updated_at
        });
    }

    public function down(): void
    {
        Schema::dropIfExists('municipios');
    }
};
ℹ️
Borrado lógico con "activo"
En SICSAL nunca borramos registros de la base de datos. En su lugar, ponemos activo = false. Esto preserva la integridad referencial y permite auditoría. Es una regla del proyecto (ver PLAN_SICSAL.md).

4 Tabla centros_salud

Crea la segunda migración:

Terminal bash 
$ php artisan make:migration create_centros_salud_table

Diseño de la tabla

Columna Tipo Descripción
PK id bigint Identificador autoincremental
FK municipio_id bigint Referencia a municipios.id
nombre varchar(150) Nombre del centro (ej: "C.S. Hornoma")
codigo_snis varchar(20) Código del SNIS (puede ser null)
subsector enum Público, Seguro Social, Privado, ONG
red_salud varchar(100) Red de salud (ej: "Red VII")
poblacion_ine integer Población INE asignada
activo boolean Borrado lógico
created_at, updated_at timestamp Automáticas

Código completo

database/migrations/XXXX_XX_XX_XXXXXX_create_centros_salud_table.php php 
<?php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    public function up(): void
    {
        Schema::create('centros_salud', function (Blueprint $table) {
            $table->id();
            $table->foreignId('municipio_id')
                  ->constrained('municipios');             // FK → municipios.id
            $table->string('nombre', 150);
            $table->string('codigo_snis', 20)
                  ->nullable();                             // puede ser null
            $table->enum('subsector', [
                'Público', 'Seguro Social',
                'Privado', 'ONG'
            ])->default('Público');
            $table->string('red_salud', 100)
                  ->nullable();
            $table->integer('poblacion_ine')
                  ->default(0);
            $table->boolean('activo')->default(true);
            $table->timestamps();
        });
    }

    public function down(): void
    {
        Schema::dropIfExists('centros_salud');
    }
};

5 Ejecutar las migraciones

Con los archivos creados, ejecuta las migraciones para que Laravel cree las tablas en MySQL:

Terminal bash 
$ php artisan migrate
INFO  Running migrations.

  2026_04_03_015720_create_municipios_table .......... 25ms DONE
  2026_04_03_015721_create_centros_salud_table ....... 32ms DONE

Verifica en MySQL que las tablas existen:

Terminal bash 
$ php artisan db:table municipios
  Column ......... Type ......... Nullable
  id ............. bigint unsigned No
  nombre ......... varchar(100) .. No
  departamento ... varchar(100) .. No
  activo ......... tinyint(1) .... No
  created_at ..... timestamp ..... Yes
  updated_at ..... timestamp ..... Yes
💡
Tabla migrations
Laravel crea una tabla oculta llamada migrations donde registra qué migraciones ya ejecutó. Así sabe que no debe ejecutar las mismas dos veces.

7 Llaves foráneas (foreign keys)

La tabla centros_salud tiene una llave foránea hacia municipios:

Llave foránea php 
$table->foreignId('municipio_id')
      ->constrained('municipios');

Esto hace dos cosas:

  1. Crea una columna municipio_id BIGINT UNSIGNED
  2. Crea un constraint FOREIGN KEY (municipio_id) REFERENCES municipios(id)
⚠️
Orden de migración importa
La tabla municipios debe existir antes que centros_salud porque la FK referencia a municipios. Por eso la migración de municipios tiene un timestamp menor.

Relación visual:

municipios
id, nombre, departamento
1 ──── N
centros_salud
id, municipio_id, nombre

Un municipio tiene muchos centros de salud. Un centro de salud pertenece a un municipio.

8 Revertir migraciones

Si necesitas deshacer las migraciones (por ejemplo, si cometiste un error en la estructura):

Terminal bash 
# Revertir el último lote de migraciones
$ php artisan migrate:rollback

# Revertir TODO y volver a migrar
$ php artisan migrate:fresh
Dropped all tables successfully.
Running migrations...

# Fresh + ejecutar seeders
$ php artisan migrate:fresh --seed
🚨
Cuidado con migrate:fresh
migrate:fresh borra todas las tablas y las recrea. Perfecto en desarrollo, nunca en producción (perderías todos los datos). En producción usa solo migrate para aplicar migraciones nuevas.
