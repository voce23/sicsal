1 ¿Qué es Spatie Permission?

spatie/laravel-permission es el paquete estándar para RBAC (Role-Based Access Control) en Laravel. Agrega:

  • Trait HasRoles al modelo User
  • Métodos: assignRole(), hasRole(), hasAnyRole(), hasPermissionTo()
  • 5 tablas pivote para roles y permisos
ℹ️
SICSAL usa solo roles, no permisos
El paquete soporta permisos granulares (can_edit_persona, can_delete_vacuna), pero SICSAL solo usa 3 roles. Las verificaciones se hacen con hasRole() directamente en Resources y traits.

2 Las 5 tablas de permisos

La migración create_permission_tables crea:

Tabla Columnas clave Propósito
roles id, name, guard_name Definición de roles
permissions id, name, guard_name Definición de permisos (vacía en SICSAL)
model_has_roles role_id, model_type, model_id Pivote: qué roles tiene cada usuario
model_has_permissions permission_id, model_type, model_id Pivote: permisos directos (no usado)
role_has_permissions permission_id, role_id Pivote: permisos por rol (no usado)

En la práctica, solo 2 tablas tienen datos: roles (3 registros) y model_has_roles (1 por usuario).

3 RolesSeeder — los 3 roles del sistema

database/seeders/RolesSeeder.php php 
use Spatie\Permission\Models\Role;

class RolesSeeder extends Seeder
{
    public function run(): void
    {
        Role::firstOrCreate(['name' => 'superadmin',  'guard_name' => 'web']);
        Role::firstOrCreate(['name' => 'admin',       'guard_name' => 'web']);
        Role::firstOrCreate(['name' => 'registrador', 'guard_name' => 'web']);
    }
}
Rol Acceso Puede ver
superadmin Todo el sistema, todos los centros Todos los Resources, selector de centro libre
admin Su centro de salud Gestión de registradores, datos de su centro
registrador Su centro de salud Registro de datos, sin gestión de usuarios
💡
firstOrCreate — idempotente
firstOrCreate busca el registro antes de crear. Puedes correr el seeder múltiples veces sin duplicar roles.

4 UserSeeder — asignación con assignRole

database/seeders/UserSeeder.php (extracto) php 
// Superadmin — acceso total
$superadmin = User::create([
    'name' => 'Super', 'apellidos' => 'Admin',
    'email' => 'superadmin@sicsal.test',
    'password' => 'password',
    'activo' => true,
]);
$superadmin->assignRole('superadmin');

// Admin — ve solo su centro
$admin = User::create([
    'name' => 'Médico', 'apellidos' => 'Hornoma',
    'email' => 'medico.hornoma@sicsal.test',
    'centro_salud_id' => 1,
    'activo' => true,
]);
$admin->assignRole('admin');

// Registrador — ve solo su centro, no gestiona usuarios
$registrador = User::create([...]);
$registrador->assignRole('registrador');

DatabaseSeeder llama los seeders en orden: RolesSeeder primero (crea roles), luego UserSeeder (asigna roles).

5 Uso en Filament: canAccess, getEloquentQuery

Ejemplo: CentroSaludResource.php php 
// Solo superadmin puede ver el Resource de Centros de Salud
public static function canAccess(): bool
{
    return auth()->user()->hasRole('superadmin');
}
Ejemplo: UserResource.php php 
// Admin solo ve registradores de su centro
public static function getEloquentQuery(): Builder
{
    $query = parent::getEloquentQuery();
    $user  = auth()->user();

    if ($user->hasRole('admin')) {
        return $query
            ->where('centro_salud_id', $user->centro_salud_id)
            ->role('registrador');  // scope de Spatie
    }
    return $query;
}
Ejemplo: UserForm.php — campo condicional php 
Select::make('centro_salud_id')
    ->visible(fn() => auth()->user()->hasRole('superadmin'))
    ->default(fn() => auth()->user()->centro_salud_id),

6 Patrón RBAC de SICSAL

superadmin
Todo el sistema, todos los centros
↓ restringe
admin
Su centro + gestión de registradores
↓ restringe
registrador
Su centro + solo registro de datos
⚠️
Sin Policies — decisión deliberada
SICSAL no tiene directorio app/Policies/. En lugar de Policies de Laravel, usa canAccess() y getEloquentQuery() de Filament. Esto simplifica el código: todo el control de acceso está en los Resources, no disperso en archivos de Policy separados.
💡
¿Cuándo agregar permisos granulares?
Si el sistema crece y necesitas control más fino (ej: "registrador puede ver pero no eliminar"), puedes agregar permisos al seeder y usar hasPermissionTo(). Spatie ya tiene la infraestructura — las tablas están creadas.

  • Sin permisos granulares ni Policies — todo vía canAccess y getEloquentQuery en Filament
