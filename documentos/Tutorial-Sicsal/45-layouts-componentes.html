1 · Visión general del sistema de layouts

SICSAL tiene tres dominios visuales completamente separados, cada uno con su layout raíz:

DominioLayout raízDelega aUso
Públicolayouts/public.blade.php—Blog, Población, C.A.I., Causas Consulta
Admin (Filament)layouts/app.blade.phpapp/sidebarPanel de administración
Authlayouts/auth.blade.phpauth/simpleLogin, Register, Forgot Password

Los layouts de admin y auth usan un patrón de delegación de un solo nivel: el layout raíz es un wrapper de 3-5 líneas que simplemente incluye un sub-layout con @include o componente Flux. Esto permite cambiar la variante visual (sidebar↔header, simple↔card↔split) sin tocar las vistas.

Flux UI: Los layouts de admin y auth usan Flux, una librería de componentes Blade para Laravel. Los <flux:sidebar>, <flux:header>, <flux:navlist> que verás son componentes Flux.

2 · Layout público — public.blade.php

El layout público es el más extenso (~115 líneas) porque define la estructura HTML completa con navbar, footer y CDNs. Es el shell para todas las páginas que ven los usuarios logueados en la interfaz pública.

Estructura general

<!-- resources/views/components/layouts/public.blade.php -->
<!DOCTYPE html>
<html lang="es" class="scroll-smooth">
<head>
    <meta charset="UTF-8">
    <title>SIMUES</title>
    <link rel="icon" href="data:image/svg+xml,<svg ...>🏥</svg>">

    <!-- CDNs externos -->
    <link ... "leaflet.css">           <!-- Mapas -->

    @vite(['resources/css/app.css', 'resources/js/app.js'])

    <!-- Fix Leaflet tiles con Tailwind v4 preflight -->
    <style>
      [x-cloak] { display: none !important; }
      .leaflet-container img { max-width: none !important; max-height: none !important; }
    </style>

    <!-- Alpine.js — interactividad del dropdown y menú móvil -->
    <script defer src="https://cdn.jsdelivr.net/npm/alpinejs@3/dist/cdn.min.js"></script>
</head>

Navbar con autenticación condicional

Alpine.js es necesario: El layout público usa directivas Alpine (x-data, x-show, @click.outside) para el dropdown de usuario y el menú hamburguesa móvil. Sin cargar Alpine.js, el dropdown queda siempre visible y no se puede cerrar. La regla [x-cloak] oculta los elementos hasta que Alpine inicialice, y el override .leaflet-container img evita que el preflight de Tailwind v4 (img { max-width:100% }) rompa los tiles del mapa Leaflet. 
<!-- Navbar sticky teal -->
<nav class="sticky top-0 z-50 bg-teal-600 shadow-lg">
  <!-- Logo "S" + SIMUES branding -->
  <a href="/">
    <span class="bg-white text-teal-700 rounded-lg font-black">S</span>
    SIMUES
  </a>

  <!-- Links visibles SOLO para usuarios autenticados -->
  @auth
    <a href="{{ route('comunidades-poblacion') }}">Comunidades</a>
    <a href="{{ route('poblacion') }}">Población</a>
    <a href="{{ route('informe-cai') }}">C.A.I.</a>
    <a href="{{ route('causas-consulta') }}">Causas Consulta</a>
    <a href="{{ route('blog') }}">Blog</a>
  @endauth

Menú de usuario con Alpine.js

<!-- Dropdown de usuario (Alpine.js) -->
@auth
<div x-data="{ drop: false }" @keydown.escape.window="drop = false">
  <button @click="drop = !drop">
    <!-- Avatar con inicial del nombre -->
    {{ substr(auth()->user()->name, 0, 1) }}
  </button>

  <div x-show="drop" @click.outside="drop = false" x-cloak>
    <!-- Link al panel admin (solo con roles) -->
    @if(auth()->user()->hasAnyRole(['superadmin','admin','registrador']))
      <a href="/admin">Panel Administración</a>
    @endif

    <!-- Logout con formulario POST -->
    <form method="POST" action="{{ route('logout') }}">
      @csrf
      <button type="submit">Cerrar sesión</button>
    </form>
  </div>
</div>
@else
  <!-- Invitados: solo Blog / Login / Registrarse -->
@endauth

Footer y scripts

<!-- Footer con branding institucional -->
<footer class="bg-teal-800 text-teal-100">
  SIMUES — Red de Salud VII Capinota — SEDES Cochabamba
</footer>

<!-- Leaflet JS + scripts apilados por cada página -->
<script src="leaflet.js"></script>
@stack('scripts')  <!-- Las vistas hijas hacen @push('scripts') -->
@stack('scripts') permite que cada página Livewire inyecte JS específico (Chart.js para pirámide poblacional, Leaflet para mapas) sin contaminar el layout base.

3 · Layout admin — app.blade.php → sidebar

El layout del panel admin usa delegación de un nivel. El archivo raíz es minimal:

<!-- resources/views/components/layouts/app.blade.php (5 líneas) -->
<x-layouts.app.sidebar>
    {{ $slot }}
</x-layouts.app.sidebar>

Toda la estructura real está en app/sidebar.blade.php (~100 líneas):

<!-- resources/views/components/layouts/app/sidebar.blade.php -->
<flux:sidebar sticky stashable class="border-r ...">
    <flux:sidebar.toggle class="lg:hidden" />

    <!-- Logo de la app -->
    <a href="{{ route('dashboard') }}">
        <x-app-logo />
    </a>

    <!-- Navegación principal -->
    <flux:navlist variant="outline">
        <flux:navlist.group heading="Plataforma" class="grid">
            <flux:navlist.item icon="home"
                href="{{ route('dashboard') }}">
                Dashboard
            </flux:navlist.item>
        </flux:navlist.group>
    </flux:navlist>

    <!-- Spacer empuja el menú de usuario al fondo -->
    <flux:spacer />

    <!-- Centro selector (solo superadmin) -->
    <x-centro-selector />

    <!-- Menú de usuario desktop -->
    <x-desktop-user-menu />
</flux:sidebar>

<!-- Header mobile (visible en pantallas pequeñas) -->
<flux:header class="lg:hidden">
    <flux:sidebar.toggle />
    <flux:spacer />
    <x-desktop-user-menu />
</flux:header>

<!-- Contenido principal -->
<flux:main>
    {{ $slot }}
</flux:main>

La variante alternativa app/header.blade.php (~86 líneas) usa <flux:header> + <flux:navbar> en vez del sidebar, con un menú horizontal. SICSAL usa la variante sidebar por defecto.

Cambiar de sidebar a header: Solo hay que editar layouts/app.blade.php para que apunte a <x-layouts.app.header> en lugar de <x-layouts.app.sidebar>. Ninguna vista hija cambia.

4 · Layout auth — auth.blade.php → simple/card/split

El layout de autenticación sigue el mismo patrón de delegación:

<!-- resources/views/components/layouts/auth.blade.php (3 líneas) -->
<x-layouts.auth.simple>
    {{ $slot }}
</x-layouts.auth.simple>

Hay tres variantes de auth layout disponibles:

VarianteDescripciónComponentes clave
auth/simple Formulario centrado, fondo limpio <x-auth-header>, logo, slot
auth/card Formulario dentro de tarjeta con sombra <flux:card>, header, <x-app-logo-icon>
auth/split Pantalla dividida: formulario izquierda, imagen/patrón derecha Grid 2 columnas, <x-placeholder-pattern>

Variante simple (activa)

<!-- layouts/auth/simple.blade.php -->
<div class="flex min-h-screen flex-col items-center
             justify-center bg-gray-50">
    <div class="w-full sm:max-w-md">
        <x-auth-header
            :title="$title ?? 'Iniciar sesión'"
            :description="$description ?? ''" />

        {{ $slot }}
    </div>
</div>

Variante split (alternativa)

<!-- layouts/auth/split.blade.php -->
<div class="grid lg:grid-cols-2 min-h-screen">
    <!-- Columna izquierda: formulario -->
    <div class="flex flex-col items-center justify-center">
        <x-auth-header ... />
        {{ $slot }}
    </div>
    <!-- Columna derecha: patrón decorativo -->
    <div class="hidden lg:flex items-center justify-center
                 bg-gray-100">
        <x-placeholder-pattern />
    </div>
</div>
SICSAL no tiene registro público: Aunque existe la ruta /register, la creación de usuarios siempre es por admin/superadmin. El layout auth existe para login, reset de contraseña y verificación de email.

6 · Centro Selector — el componente superadmin

El componente más importante para el control de acceso visual. Solo aparece para el superadmin y permite cambiar de centro de salud en tiempo real:

<!-- resources/views/components/centro-selector.blade.php -->
@if(auth()->user()?->hasRole('superadmin'))
  <div class="px-3 py-2">
    <label class="block text-xs font-medium text-amber-600
                   uppercase tracking-wider mb-1">
      Centro de Salud
    </label>

    <select
      wire:model.live="centroSaludId"
      class="w-full rounded-md border-amber-300
             bg-amber-50 text-amber-900 text-sm
             focus:border-amber-500 focus:ring-amber-500">

      @foreach($this->centrosDisponibles as $centro)
        <option value="{{ $centro->id }}">
          {{ $centro->nombre }}
        </option>
      @endforeach

    </select>
  </div>
@endif

¿Cómo funciona?

  1. Visibilidad: @if(auth()->user()?->hasRole('superadmin')) — solo superadmin ve el selector.
  2. Binding: wire:model.live="centroSaludId" — enlace bidireccional con Livewire. Cuando cambia el valor, Livewire actualiza la propiedad $centroSaludId en el componente padre.
  3. Opciones: $this->centrosDisponibles viene del trait HasCentroSelector que vimos en la Lección 19.
  4. Estilo amber: Diferenciación visual intencional. El amarillo/amber indica "modo superadmin" — una pista visual de que estás viendo datos de un centro ajeno.
Seguridad: El selector es SOLO visual. La validación real de permisos está en los modelos (scopes centro_salud_id), las policies y el trait HasCentroSelector. Nunca confíes solo en ocultar un elemento HTML.

<!-- Mensaje tipo "Guardado!" que aparece y desaparece -->

8 · Rutas de Settings — routes/settings.php

Las páginas de configuración de usuario tienen su propio archivo de rutas:

// routes/settings.php
use Illuminate\Support\Facades\Route;
use Livewire\Volt\Volt;

Route::middleware(['auth'])->group(function () {
    Route::redirect('/settings', '/settings/profile');

    Volt::route('/settings/profile',  'settings.profile')
         ->name('settings.profile');

    Volt::route('/settings/security', 'settings.security')
         ->name('settings.security');

    Volt::route('/settings/appearance', 'settings.appearance')
         ->name('settings.appearance');
});

Puntos clave

  • Volt::route(): Registra rutas que apuntan directamente a componentes Volt (single-file Livewire) sin necesidad de un controlador.
  • Middleware auth: Todas las settings requieren autenticación.
  • Redirect /settings → /settings/profile: La URL base redirige a la primera pestaña.

9 · Páginas de configuración con Volt

Las páginas de settings usan Volt, que permite combinar lógica PHP y vista Blade en un solo archivo .blade.php.

Layout de settings (la barra lateral)

<!-- resources/views/pages/settings/layout.blade.php -->
<div class="flex flex-col lg:flex-row gap-8">
    <!-- Navegación lateral de settings -->
    <flux:navlist class="lg:w-48">
        <flux:navlist.item href="{{ route('settings.profile') }}"
            :current="request()->routeIs('settings.profile')"
            icon="user" wire:navigate>
            Perfil
        </flux:navlist.item>

        <flux:navlist.item href="{{ route('settings.security') }}"
            :current="request()->routeIs('settings.security')"
            icon="lock-closed" wire:navigate>
            Seguridad
        </flux:navlist.item>

        <flux:navlist.item href="{{ route('settings.appearance') }}"
            :current="request()->routeIs('settings.appearance')"
            icon="swatch" wire:navigate>
            Apariencia
        </flux:navlist.item>
    </flux:navlist>

    <!-- Contenido de la página actual -->
    <div class="flex-1">
        {{ $slot }}
    </div>
</div>

Perfil (settings/profile.blade.php) — Volt

<!-- Ejemplo de componente Volt: PHP + Blade en un archivo -->
<?php
use Livewire\Volt\Component;
use Illuminate\Validation\Rule;
use App\Models\User;

new class extends Component
{
    public string $name = '';
    public string $email = '';

    public function mount(): void
    {
        $this->name  = auth()->user()->name;
        $this->email = auth()->user()->email;
    }

    public function updateProfileInformation(): void
    {
        $validated = $this->validate([
            'name'  => ['required', 'string', 'max:255'],
            'email' => ['required', 'email', 'max:255',
                        Rule::unique(User::class)
                            ->ignore(auth()->id())],
        ]);

        auth()->user()->fill($validated);

        if (auth()->user()->isDirty('email')) {
            auth()->user()->email_verified_at = null;
        }

        auth()->user()->save();

        $this->dispatch('profile-updated');
    }
};
?>

<!-- Parte Blade del mismo archivo -->
<section>
    @include('pages.settings.layout')
    <form wire:submit="updateProfileInformation">
        <flux:input wire:model="name" label="Nombre" />
        <flux:input wire:model="email" label="Email" />

        <flux:button type="submit">Guardar</flux:button>

        <x-action-message on="profile-updated">
            Guardado.
        </x-action-message>
    </form>
</section>

Seguridad (settings/security.blade.php)

Permite cambiar la contraseña. Usa Hash::check() para validar la contraseña actual antes de aceptar la nueva:

public function updatePassword(): void
{
    $validated = $this->validate([
        'current_password' => ['required', 'current_password'],
        'password'         => ['required', 'confirmed',
                              Rules\Password::defaults()],
    ]);

    auth()->user()->update([
        'password' => Hash::make($validated['password']),
    ]);
}

Apariencia (settings/appearance.blade.php)

Usa JavaScript puro (<script> en la página) para alternar entre tema claro, oscuro y sistema. No tiene lógica PHP — es 100% cliente.

Eliminar cuenta

El formulario delete-user-form requiere confirmación de contraseña antes de eliminar la cuenta del usuario.

Volt = PHP inline: En archivos Volt, la clase PHP se define con new class extends Component { ... } al inicio del .blade.php. No hay archivo PHP separado. Es ideal para páginas pequeñas con lógica simple como settings.

10 · El patrón de delegación en cascada

Resumen del patrón arquitectónico que conecta todos los layouts:

Dominio Público
┌──────────────────────────────────────────────┐
│  layouts/public.blade.php  (115 líneas)     │
│  ├── Navbar sticky teal + @auth condicional  │
│  ├── {{ $slot }}  ← Livewire pages            │
│  ├── Footer SIMUES                           │
│  └── @stack('scripts')                       │
└──────────────────────────────────────────────┘

Dominio Admin
┌──────────────────────┐     ┌─────────────────────────┐
│  layouts/app.blade   │────▶│  app/sidebar.blade      │
│  (5 líneas)          │     │  ├── flux:sidebar        │
│  Delega a sidebar    │     │  ├── x-centro-selector   │
└──────────────────────┘     │  ├── x-desktop-user-menu │
       Alternativa:          │  └── flux:main → $slot   │
  app/header.blade           └─────────────────────────┘
  (86 líneas, navbar)

Dominio Auth
┌──────────────────────┐     ┌──────────────────────┐
│  layouts/auth.blade  │────▶│  auth/simple.blade   │
│  (3 líneas)          │     │  Formulario centrado  │
│  Delega a simple     │     └──────────────────────┘
└──────────────────────┘     Alternativas:
                              auth/card.blade   (con tarjeta)
                              auth/split.blade  (pantalla dividida)

¿Por qué este patrón?

  • Intercambiabilidad: Cambiar sidebar↔header o simple↔card↔split requiere editar UNA línea en el layout raíz.
  • Separación de responsabilidades: El layout público maneja CDNs y navbar pública; el admin maneja Flux y permisos; el auth es minimal.
  • Componentes compartidos: <x-app-logo>, <x-centro-selector> y <x-desktop-user-menu> se usan en múltiples layouts sin duplicación.
  • Stacks para JS: @stack('scripts') en el layout público permite que cada página inyecte solo el JavaScript que necesita.

Resumen de la lección

  • SICSAL tiene 3 dominios visuales: público (teal, navbar, CDNs), admin (Flux sidebar) y auth (minimalista).
  • Los layouts admin y auth usan delegación de un nivel — el raíz es un wrapper de 3-5 líneas que incluye una variante intercambiable.
  • El layout público es el más completo: navbar con @auth condicional, dropdown Alpine.js, footer institucional, @stack('scripts').
  • El centro-selector es exclusivo del superadmin, con wire:model.live y estilo amber diferenciado.
  • Las settings pages usan Volt (PHP + Blade en un archivo) con rutas Volt::route() — perfil, seguridad, apariencia y eliminar cuenta.
  • El componente action-message demuestra la integración Alpine.js + Livewire para feedback temporal.
