1 El patrón Prest*: modelo mínimo, máxima utilidad

Los 16 modelos Prest* son los más simples del proyecto. Todos comparten esta estructura:

Plantilla genérica Prest* php 
class PrestXxx extends Model
{
    protected $table = 'prest_xxx';

    protected $fillable = [
        'centro_salud_id', 'mes', 'anio',
        // + dimensión (tipo, indicador, grupo_etareo...)
        // + contadores (cantidad, masculino/femenino, dentro/fuera...)
    ];

    protected function casts(): array
    {
        return ['cantidad' => 'integer'];
    }

    public function centroSalud(): BelongsTo
    {
        return $this->belongsTo(CentroSalud::class);
    }
}
💡
¿Por qué son tan simples?
Los modelos Prest* no tienen scopes, accessors ni lifecycle hooks. Su complejidad está en la migración (constraints UNIQUE, enums) y en las vistas/exports que los consumen. El modelo solo es el puente entre la tabla y el código PHP.

Diferencias clave entre los 16 modelos:

M/F primera/nueva/repetida PrestConsultaExterna primera_m, primera_f, nueva_m, nueva_f, repetida_m, repetida_f
M/F nuevos/repetidos PrestCrecimiento nuevos_m, nuevos_f, repetidos_m, repetidos_f

ℹ️
PrestIle — el más minimalista
PrestIle ni siquiera define casts() ni relación centroSalud() — es un modelo de solo 7 líneas. Funciona porque Eloquent asume valores por defecto sensatos. Sin embargo, agregar la relación es recomendable para consistencia.

5 MetaIne — poblaciones de referencia

app/Models/MetaIne.php php 
class MetaIne extends Model
{
    protected $table = 'metas_ine';

    protected $fillable = [
        'centro_salud_id', 'anio', 'grupo_etareo',
        'sexo', 'cantidad',
    ];

    protected function casts(): array
    {
        return ['cantidad' => 'integer'];
    }

    public function centroSalud(): BelongsTo
    {
        return $this->belongsTo(CentroSalud::class);
    }
}

Este modelo es el denominador de todas las fórmulas de cobertura. Sin él, no podemos calcular qué porcentaje de la población meta fue atendida:

Ejemplo — calcular cobertura php 
// Meta INE: menores de 1 año en el centro
$meta = MetaIne::where('centro_salud_id', $centroId)
    ->where('anio', 2026)
    ->where('grupo_etareo', 'menor_1')
    ->where('sexo', 'ambos')
    ->value('cantidad');  // → ej: 50

// Prestación: vacunas BCG aplicadas
$vacunados = PrestVacuna::where(...)
    ->sum(DB::raw('dentro_m + dentro_f + fuera_m + fuera_f'));

// Cobertura = (vacunados / meta) × 100
$cobertura = $meta > 0
    ? round($vacunados / $meta * 100, 1)
    : 0;

6 JustificacionCero — cero justificado

app/Models/JustificacionCero.php php 
class JustificacionCero extends Model
{
    protected $table = 'justificaciones_cero';

    protected $fillable = [
        'centro_salud_id', 'mes', 'anio', 'indicador',
        'motivo', 'detalle', 'registrado_por',
    ];

    public function centroSalud(): BelongsTo
    {
        return $this->belongsTo(CentroSalud::class);
    }

    public function registradoPor(): BelongsTo
    {
        return $this->belongsTo(User::class, 'registrado_por');
    }
}
⚠️
Regla de negocio fundamental
Si un indicador clave (control_prenatal, partos, vacunación, etc.) tiene valor cero en un mes, el sistema obliga al registrador a justificar por qué. Los 5 motivos posibles están definidos como enum en la migración. Este modelo almacena esa justificación.

7 MesCerrado — control de período

app/Models/MesCerrado.php php 
class MesCerrado extends Model
{
    protected $table = 'meses_cerrados';

    protected $fillable = [
        'centro_salud_id', 'mes', 'anio',
        'cerrado_por', 'fecha_cierre',
        'reabierto_por', 'fecha_reapertura',
    ];

    protected function casts(): array
    {
        return [
            'fecha_cierre'    => 'datetime',
            'fecha_reapertura' => 'datetime',
        ];
    }

    public function centroSalud(): BelongsTo { ... }
    public function cerradoPor(): BelongsTo
    {
        return $this->belongsTo(User::class, 'cerrado_por');
    }
    public function reabiertoPor(): BelongsTo
    {
        return $this->belongsTo(User::class, 'reabierto_por');
    }
}

Este modelo tiene 2 relaciones a User con FKs diferentes: cerrado_por y reabierto_por. Nótese el uso de 'datetime' en vez de 'date' — aquí sí importa la hora exacta del cierre.

ℹ️
Flujo de cierre/reapertura
Admin cierra un mes → se graban cerrado_por y fecha_cierre.
Si hay error, superadmin reabre → se graban reabierto_por y fecha_reapertura.
