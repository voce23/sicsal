1 PersonaResource — navegación y filtrado

app/Filament/Resources/Personas/PersonaResource.php
class PersonaResource extends Resource
{
    protected static ?string $model = Persona::class;
    protected static string|BackedEnum|null $navigationIcon
        = Heroicon::OutlinedUserGroup;
    protected static string|\UnitEnum|null $navigationGroup
        = 'Padrón Poblacional';
    protected static ?string $navigationLabel = 'Padrón de Personas';
    protected static ?string $modelLabel      = 'Persona';
    protected static ?string $pluralModelLabel = 'Personas';
    protected static ?int    $navigationSort   = 1;
    protected static ?string $recordTitleAttribute = 'nombres';

    // Filtrado por centro (igual que ComunidadResource)
    public static function getEloquentQuery(): Builder
    {
        $query = parent::getEloquentQuery();
        $user = auth()->user();
        if ($user && !$user->hasRole('superadmin')) {
            $query->where('centro_salud_id', $user->centro_salud_id);
        }
        return $query;
    }

    // Delegación a Form y Table separados
    public static function form(Schema $schema): Schema
    {
        return PersonaForm::configure($schema);
    }

    public static function table(Table $table): Table
    {
        return PersonasTable::configure($table);
    }

    public static function getPages(): array
    {
        return [
            'index'  => ListPersonas::route('/'),
            'create' => CreatePersona::route('/create'),
            'edit'   => EditPersona::route('/{record}/edit'),
        ];
    }
}
Grupo de navegación: Padrón Poblacional

PersonaResource no usa canAccess() — cualquier usuario autenticado puede acceder. Pero getEloquentQuery() garantiza que admin/registrador solo ven personas de su propio centro.

Radio::make() — alternativa a Select

Para campos con solo 2-3 opciones, Radio con ->inline() es más ergonómico que un Select. El usuario ve las opciones sin tener que abrir un dropdown.

Sección 2: Seguro y ubicación

PersonaForm.php — Sección seguro y ubicación
Section::make('Seguro y ubicación')
    ->schema([
        Select::make('tipo_seguro')
            ->options([
                'SUS'     => 'SUS (Seguro Universal de Salud)',
                'privado' => 'Privado',
                'ninguno' => 'Ninguno',
            ])
            ->default('ninguno')->required()->label('Tipo de seguro'),
        Select::make('comunidad_id')
            ->relationship(
                'comunidad', 'nombre',
                fn ($query) => $query->where(
                    'centro_salud_id',
                    auth()->user()?->centro_salud_id
                )
            )
            ->searchable()->preload()->required()
            ->label('Comunidad'),
        DatePicker::make('fecha_registro')
            ->required()->default(today())->label('Fecha de registro'),
    ])
    ->columns(2),
relationship() con filtro — seguridad clave

El tercer argumento de relationship() recibe un closure que filtra las opciones del Select. Aquí se limitan las comunidades a las del centro del usuario. Sin esto, un admin podría asignar personas a comunidades de otros centros.

Sección 3: Estado migratorio — campos condicionales

PersonaForm.php — Sección estado migratorio
Section::make('Estado migratorio')
    ->schema([
        Select::make('estado')
            ->options([
                'residente' => 'Residente',
                'temporal'  => 'Temporal',
                'migrado'   => 'Migrado',
                'fallecido' => 'Fallecido',
            ])
            ->default('residente')->required()
            ->live()
            ->label('Estado'),
        TextInput::make('destino_migracion')
            ->maxLength(150)
            ->label('Destino de migración')
            ->hidden(fn ($get) => $get('estado') !== 'migrado'),
        DatePicker::make('fecha_migracion')
            ->label('Fecha de migración')
            ->hidden(fn ($get) => $get('estado') !== 'migrado'),
    ])
    ->columns(2),
live() + hidden() — formularios dinámicos

->live() en el campo estado hace que Filament envíe una petición AJAX cada vez que el valor cambia. Los campos destino_migracion y fecha_migracion usan ->hidden(fn ($get) => ...) para ocultarse cuando el estado no es "migrado". El $get closure accede al valor actual de otro campo del formulario.

Sección 4: Verificación del padrón — afterStateUpdated

PersonaForm.php — Sección verificación
Section::make('Verificación del padrón')
    ->description('Confirme si la persona fue verificada...')
    ->schema([
        Toggle::make('verificado')
            ->label('Persona verificada')
            ->helperText('Active si la presencia fue confirmada en la visita.')
            ->live()
            ->afterStateUpdated(function ($state, callable $set) {
                if ($state) {
                    $set('fecha_verificacion', today()->format('Y-m-d'));
                    $set('verificado_por', auth()->id());
                }
            }),
        DatePicker::make('fecha_verificacion')
            ->label('Fecha de verificación')
            ->hidden(fn ($get) => !$get('verificado')),
    ])
    ->columns(2),
afterStateUpdated — auto-llenar otros campos

Cuando el toggle de verificado se activa, SICSAL auto-llena:

  • fecha_verificacion = hoy
  • verificado_por = ID del usuario actual

$set(campo, valor) modifica el valor de otro campo del formulario. $get(campo) lee el valor. Juntos permiten formularios con lógica compleja sin JavaScript.

->columns([
    // ① Icono de verificación con tooltip

        ->label('Ver.')->boolean()

        ->falseIcon('heroicon-m-clock')

        ->sortable()->label('Apellidos'),

    TextColumn::make('comunidad.nombre')
        ->searchable()->sortable()->label('Comunidad'),

    // ⑤ Accessor del modelo (no sortable)

        ->getStateUsing(fn ($record) => $record->edad . ' años')
        ->sortable(false)->label('Edad'),

    // ⑥ Badge con color por valor
    TextColumn::make('sexo')
        ->badge()
        ->color(fn (string $state) => match ($state) {
            'M' => 'info',
            'F' => 'danger',
            default => 'gray',
        })
        ->formatStateUsing(fn (string $state) =>
            $state === 'M' ? 'Masculino' : 'Femenino')
        ->label('Sexo'),

    // ⑦ Seguro (badge simple)
    TextColumn::make('tipo_seguro')->badge()->label('Seguro'),

    // ⑧ Estado con 4 colores
    TextColumn::make('estado')
        ->badge()
        ->color(fn (string $state) => match ($state) {
            'residente' => 'success',
            'temporal'  => 'warning',
            'migrado'   => 'danger',
            'fallecido' => 'gray',
            default     => 'gray',
        })
        ->formatStateUsing(fn (string $state) => match ($state) {
            'residente' => 'Residente',
            'temporal'  => 'Temporal',
            'migrado'   => 'Migrado',
            'fallecido' => 'Fallecido',
            default     => $state,
        })->label('Estado'),

    // ⑨ y ⑩ Columnas toggleables (ocultas por defecto)
    TextColumn::make('fecha_verificacion')
        ->date('d/m/Y')->sortable()
        ->toggleable(isToggledHiddenByDefault: true),

        ->date('d/m/Y')->sortable()
        ->toggleable(isToggledHiddenByDefault: true),

4 Filtros: 5 filtros combinables

PersonasTable.php — filtros
->filters([
    SelectFilter::make('comunidad_id')
        ->relationship('comunidad', 'nombre')
        ->searchable()->preload()->label('Comunidad'),
    SelectFilter::make('sexo')
        ->options(['M' => 'Masculino', 'F' => 'Femenino'])
        ->label('Sexo'),
    SelectFilter::make('tipo_seguro')
        ->options(['SUS' => 'SUS', 'privado' => 'Privado', 'ninguno' => 'Ninguno'])
        ->label('Tipo de seguro'),
    SelectFilter::make('estado')
        ->options([
            'residente' => 'Residente', 'temporal' => 'Temporal',
            'migrado' => 'Migrado', 'fallecido' => 'Fallecido',
        ])->label('Estado'),
    TernaryFilter::make('verificado')
        ->label('Verificación')
        ->trueLabel('Solo verificados')
        ->falseLabel('Sin verificar')
        ->nullable(),
])
TernaryFilter — tres estados

A diferencia de SelectFilter (que filtra por un valor específico), TernaryFilter ofrece tres opciones: Todos, Solo true, Solo false. Ideal para campos booleanos como verificado.

5 Acciones: rápida + 3 masivas

Acción rápida por registro: Verificar

PersonasTable.php — acción rápida de verificación
->recordActions([
    Action::make('verificar_rapido')
        ->label('Verificar')
        ->icon('heroicon-m-check-badge')
        ->color('success')
        ->visible(fn ($record) =>
            !$record->verificado
            && in_array($record->estado, ['residente', 'temporal'])
        )
        ->requiresConfirmation()
        ->modalHeading('Confirmar verificación')
        ->modalDescription(fn ($record) =>
            "¿Confirma que {$record->nombre_completo} sigue residiendo?")
        ->action(function ($record) {
            $record->update([
                'verificado'         => true,
                'fecha_verificacion' => today(),
                'verificado_por'     => auth()->id(),
            ]);
        }),
    EditAction::make(),
])
visible() — condicionar acciones

El botón "Verificar" solo aparece si la persona no está verificada Y su estado es "residente" o "temporal". No tiene sentido verificar a alguien ya marcado como migrado o fallecido.

3 Acciones masivas (BulkActions)

PersonasTable.php — acciones masivas
->toolbarActions([
    BulkActionGroup::make([
        // ① Verificar masivo
        BulkAction::make('verificar_masivo')
            ->label('Marcar como verificadas')
            ->icon('heroicon-o-check-badge')->color('success')
            ->requiresConfirmation()
            ->action(function ($records) {
                $records->each(fn ($r) => $r->update([...]));
            })->deselectRecordsAfterCompletion(),

        // ② Migración masiva (con formulario propio)
        BulkAction::make('registrar_migracion')
            ->label('Registrar como migrados')
            ->icon('heroicon-o-arrow-right-circle')->color('warning')
            ->form([
                TextInput::make('destino_migracion')
                    ->label('Destino (ciudad / país)')
                    ->maxLength(150),
                DatePicker::make('fecha_migracion')
                    ->default(today())->required(),
            ])
            ->action(function ($records, array $data) {
                $records->each(fn ($r) => $r->update([
                    'estado'            => 'migrado',
                    'destino_migracion' => $data['destino_migracion'],
                    'fecha_migracion'   => $data['fecha_migracion'],
                    // también marca como verificado
                ]));
            })->deselectRecordsAfterCompletion(),

        // ③ Fallecimiento masivo
        BulkAction::make('registrar_fallecido')
            ->label('Registrar como fallecidos')
            ->icon('heroicon-o-x-circle')->color('danger')
            ->requiresConfirmation()
            ->action(function ($records) {
                $records->each(fn ($r) => $r->update([
                    'estado' => 'fallecido', ...
                ]));
            })->deselectRecordsAfterCompletion(),

        DeleteBulkAction::make(),
    ]),
])
BulkAction con formulario propio

La acción registrar_migracion incluye ->form([...]) — esto abre un modal con campos adicionales antes de ejecutar la acción. El usuario selecciona varias personas, hace clic en "Registrar como migrados", y SICSAL le pide el destino y la fecha. Todas las personas seleccionadas se actualizan con los mismos datos.

6 Pages: CreatePersona con mutateFormData

app/Filament/Resources/Personas/Pages/CreatePersona.php
class CreatePersona extends CreateRecord
{
    protected static string $resource = PersonaResource::class;

    protected function mutateFormDataBeforeCreate(array $data): array
    {
        $data['centro_salud_id'] = auth()->user()->centro_salud_id;
        return $data;
    }
}
mutateFormDataBeforeCreate — inyección segura del centro

El campo centro_salud_id no aparece en el formulario (no es un campo editable). Se inyecta automáticamente desde el usuario autenticado justo antes de guardar. Esto es más seguro que un campo hidden o disabled — el usuario no puede manipularlo.

Las otras dos Pages son idénticas al patrón estándar:

  • EditPersona — extends EditRecord + DeleteAction
  • ListPersonas — extends ListRecords + CreateAction

7 Resumen

Lo que aprendimos

  • Radio::make() con ->inline() — para opciones de 2-3 valores
  • relationship() con filtro — tercer argumento closure filtra las opciones del Select
  • live() + hidden($get) — campos que aparecen/desaparecen según otro campo
  • afterStateUpdated($set) — auto-llenar campos cuando otro cambia
  • searchable(isIndividual: true, isGlobal: true) — doble búsqueda
  • getStateUsing() — columnas calculadas desde accessors del modelo
  • badge() + color(match()) — chips de colores por valor
  • toggleable() — columnas ocultas que el usuario puede activar
  • Action::make() con visible() — acciones condicionales por registro
  • BulkAction con form() — acciones masivas con formulario modal
  • mutateFormDataBeforeCreate() — inyectar datos del servidor antes de guardar
