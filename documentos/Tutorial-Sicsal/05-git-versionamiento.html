1 ¿Qué es Git?

Git es un sistema de control de versiones. Registra cada cambio que haces en tus archivos, como un historial de "guardar partida" en un videojuego.

Sin Git Con Git
Rompes algo y no puedes volver atrás Vuelves a cualquier punto anterior con git checkout
"sicsal-v1", "sicsal-v2-final", "sicsal-v2-final-final" Un solo proyecto con historial completo de cambios
No sabes qué cambió ni cuándo git log muestra quién cambió qué y por qué
ℹ️
Git vs GitHub
Git = herramienta local que registra cambios. GitHub = plataforma web para alojar repositorios remotos y colaborar. Puedes usar Git sin GitHub, pero GitHub necesita Git.

2 Inicializar el repositorio

Laravel ya inicializa Git cuando creas el proyecto con Composer. Verifica:

Terminal (dentro de C:\laragon\www\sicsal) bash 
$ git status
On branch main
nothing to commit, working tree clean

Si ves "fatal: not a git repository", inicialízalo manualmente:

Terminal bash 
$ git init
Initialized empty Git repository in C:/laragon/www/sicsal/.git/

# Configura tu identidad (solo la primera vez)
$ git config user.name "Tu Nombre"
$ git config user.email "tu@email.com"

3 El archivo .gitignore

El .gitignore lista archivos y carpetas que Git debe ignorar (no rastrear). Laravel ya trae uno bien configurado:

.gitignore gitignore 
/.phpunit.cache
/node_modules
/public/build
/public/hot
/public/storage
/storage/*.key
/storage/pail
/vendor
.env
.env.backup
.env.production
.phpactor.json
.phpunit.result.cache
Homestead.json
Homestead.yaml
npm-debug.log
yarn-error.log
/auth.json
/.fleet
/.idea
/.nova
/.vscode
/.zed

¿Por qué se ignoran estos archivos?

Ignorado Razón
/vendor Miles de archivos de terceros. Se regenera con composer install
/node_modules Dependencias npm. Se regenera con npm install
.env CRÍTICO: contiene contraseñas y APP_KEY. Nunca al repo
/public/build Assets compilados. Se regenera con npm run build
/storage/*.key Claves de encriptación del almacenamiento
/.vscode Configuración personal del editor
🚨
Regla de seguridad #1
Nunca subas .env al repositorio. Si accidentalmente lo haces, aunque lo borres después, el historial de Git lo conserva. Cualquiera que clone el repo podría ver tus contraseñas. Si pasa, cambia inmediatamente todas las credenciales.

4 Primer commit

Un commit es una "foto" del estado de tu proyecto en un momento dado. Hagamos nuestro primer commit con todo lo que llevarmos hasta ahora:

Terminal bash 
# Ver qué archivos hay sin rastrear
$ git status
Untracked files:
  app/
  bootstrap/
  composer.json
  ...

# Agregar TODOS los archivos al staging
$ git add .

# Hacer el commit con un mensaje descriptivo
$ git commit -m "Proyecto SICSAL inicial: Laravel 13 + Filament 5 + Livewire 4"
[main (root-commit) abc1234] Proyecto SICSAL inicial: Laravel 13 + Filament 5 + Livewire 4
 89 files changed, 12345 insertions(+)

Ahora tu proyecto tiene un punto de referencia. Si algo sale mal, puedes volver a este commit.

1
Editar archivos (working directory)
git add →
2
Staging area (listos para commit)
git commit →
3
Repository (historial permanente)

5 Comandos básicos de Git

Los comandos que usarás en el día a día:

Comando Qué hace Cuándo usarlo
git status Muestra archivos modificados/nuevos Antes de cada commit
git add . Agrega todos los cambios al staging Cuando estás listo para hacer commit
git add archivo.php Agrega un archivo específico Para commits selectivos
git commit -m "msg" Crea un commit con mensaje Al final de cada lección
git log --oneline Historial resumido Para ver commits anteriores
git diff Muestra los cambios no staged Para revisar qué cambiaste
git checkout -- archivo Descarta cambios de un archivo Cuando rompes algo y quieres revertir
git stash Guarda cambios temporalmente Para cambiar de tarea sin hacer commit

6 Buenas prácticas para commits

T recomiendo hacer un commit al final de cada lección. Ejemplos de mensajes claros:

Ejemplos de mensajes de commit git 
# ✅ Buenos mensajes - describen QUÉ se hizo
"Crear migraciones: municipios, centros_salud, comunidades"
"Agregar modelo Persona con scopes y accessors"
"Configurar Spatie Permission: roles y permisos"
"Crear PersonaResource con formulario y tabla"

# ❌ Malos mensajes - no dicen nada útil
"cambios"
"fix"
"asdf"
"WIP"
💡
Regla del buen commit
Un buen mensaje responde: "Si aplico este commit, va a..." + tu mensaje. Ejemplo: "Si aplico este commit, va a crear migraciones para municipios y centros de salud".

7 Git con VS Code

VS Code tiene Git integrado. No necesitas la terminal para todo:

1
Ver cambios: Haz clic en el icono de Git en la barra lateral izquierda (rama con puntos). Verás la lista de archivos modificados.
2
Stage (add): Pasa el mouse sobre un archivo y haz clic en el + para agregarlo al staging.
3
Commit: Escribe el mensaje en el campo superior y presiona Ctrl+Enter (o el botón ✓).
4
Ver diff: Haz clic en un archivo modificado para ver las diferencias lado a lado (rojo = eliminado, verde = agregado).
ℹ️
Atajos útiles
Ctrl+Shift+G — Abrir panel Git
Ctrl+Shift+` — Abrir terminal integrada
Los indicadores en la barra inferior muestran la rama actual y el número de cambios pendientes.

✅ Resumen de la Lección 05

  • Git registra cada cambio como un historial de "guardar partida"
  • .gitignore evita subir vendor/, node_modules/ y .env al repositorio

  • Flujo: editar → git add . → git commit -m "mensaje"
  • Mensajes de commit descriptivos: qué se hizo, no "fix" o "cambios"
  • VS Code tiene Git integrado con interfaz gráfica
  • Haz un commit al final de cada lección del tutorial
