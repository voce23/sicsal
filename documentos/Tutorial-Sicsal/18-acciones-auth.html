1 El patrón Action en Laravel

Un Action es una clase con un solo método público que encapsula una operación de negocio. Fortify usa este patrón para que tú controles cómo se crean usuarios y se resetean passwords — sin tocar los controllers internos de Fortify.

2 CreateNewUser — registro con aprobación

app/Actions/Fortify/CreateNewUser.php php 
use App\Concerns\PasswordValidationRules;
use Laravel\Fortify\Contracts\CreatesNewUsers;

class CreateNewUser implements CreatesNewUsers
{
    use PasswordValidationRules;

    public function create(array $input): User
    {
        Validator::make($input, [
            'name'            => ['required', 'string', 'max:100'],
            'apellidos'       => ['required', 'string', 'max:100'],
            'email'           => ['required', 'string', 'email', 'max:255',
                                  Rule::unique(User::class)],
            'centro_salud_id' => ['required',
                                  Rule::exists(CentroSalud::class, 'id')],
            'password'        => $this->passwordRules(),
        ])->validate();

        return User::create([
            'name'            => $input['name'],
            'apellidos'       => $input['apellidos'],
            'email'           => $input['email'],
            'centro_salud_id' => $input['centro_salud_id'],
            'password'        => $input['password'], // hashed cast lo hashea automáticamente
            'activo'          => false,             // ← clave: nace inactivo
        ]);
    }
}
⚠️
activo = false — la regla de oro
Todo usuario nuevo nace con activo = false. Un admin o superadmin debe activarlo desde el panel Filament. Esto es un requisito del SNIS: no puede haber acceso libre al sistema de salud.

Puntos clave de la validación:

  • Rule::unique(User::class) — verifica que el email no exista
  • Rule::exists(CentroSalud::class, 'id') — el centro de salud debe existir
  • $this->passwordRules() — reglas del trait PasswordValidationRules (lección 20)

3 ResetUserPassword — reset seguro

app/Actions/Fortify/ResetUserPassword.php php 
class ResetUserPassword implements ResetsUserPasswords
{
    use PasswordValidationRules;

    public function reset(User $user, array $input): void
    {
        Validator::make($input, [
            'password' => $this->passwordRules(),
        ])->validate();

        $user->forceFill([
            'password' => $input['password'],
        ])->save();
    }
}
💡
¿Por qué forceFill y no fill?
forceFill() ignora $fillable. Aunque password sí está en $fillable de User, Fortify usa forceFill() por convención para evitar problemas si alguien lo quita. El cast 'hashed' sigue aplicándose normalmente.

4 RequireAprobado — middleware guardián

app/Http/Middleware/RequireAprobado.php php 
class RequireAprobado
{
    public function handle(Request $request, Closure $next): Response
    {
        if ($request->user() && ! $request->user()->activo) {
            return redirect()->route('pendiente');
        }

        return $next($request);
    }
}

Este middleware se aplica a las rutas protegidas. Verifica dos cosas:

  1. ¿Hay un usuario autenticado? ($request->user())
  2. ¿Su campo activo es false?

Si ambas son verdaderas → redirige a /pendiente. Si no → permite pasar.

ℹ️
Complementa a canAccessPanel
canAccessPanel() en el modelo User bloquea el acceso al panel Filament. RequireAprobado bloquea el acceso a rutas web normales. Juntos, garantizan que un usuario inactivo no accede a nada.
