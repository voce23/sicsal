1 UserResource — query scoping avanzado

app/Filament/Resources/Users/UserResource.php
class UserResource extends Resource
{
    protected static ?string $model = User::class;
    protected static string|BackedEnum|null $navigationIcon
        = Heroicon::OutlinedUsers;
    protected static string|\UnitEnum|null $navigationGroup
        = 'Configuración';
    protected static ?int $navigationSort = 4;

    public static function canAccess(): bool
    {
        return auth()->user()?->hasAnyRole(['superadmin', 'admin']) ?? false;
    }

    public static function getEloquentQuery(): Builder
    {
        $query = parent::getEloquentQuery();
        $user  = auth()->user();
        if ($user && $user->hasRole('admin')) {
            // Admin solo ve registradores de su centro
            $query->where('centro_salud_id', $user->centro_salud_id)
                  ->whereHas('roles',
                      fn (Builder $q) => $q->where('name', 'registrador')
                  );
        }
        return $query;
    }
}
Query scoping doble — centro + rol

A diferencia de otros Resources que solo filtran por centro_salud_id, UserResource agrega un segundo filtro por rol:

  • Superadmin: ve TODOS los usuarios de todos los centros
  • Admin: solo ve registradores de su propio centro (no puede ver a otros admins ni al superadmin)

whereHas('roles', fn ($q) => ...) filtra por relación polimórfica de Spatie Permission.

3 Password condicional: crear vs editar

UserForm.php — campo password
Section::make('Acceso al sistema')
    ->schema([
        TextInput::make('usuario')
            ->required()->maxLength(50)
            ->unique(ignoreRecord: true)
            ->label('Nombre de usuario'),

        TextInput::make('password')
            ->password()->revealable()
            ->required(fn (string $operation) =>
                $operation === 'create')
            ->dehydrated(fn (?string $state) =>
                filled($state))
            ->minLength(8)
            ->label(fn (string $operation) =>
                $operation === 'create'
                    ? 'Contraseña'
                    : 'Nueva contraseña (dejar vacío para no cambiar)'
            ),

        // ... roles y centro_salud_id ...

        Toggle::make('activo')->default(true),
    ])->columns(2),
Tres trucos del campo password
  1. required(fn ($operation) => $operation === 'create') — obligatorio al crear, opcional al editar
  2. dehydrated(fn ($state) => filled($state)) — solo se envía al servidor si tiene valor. Si está vacío al editar, NO se sobreescribe la contraseña existente
  3. label(fn ($operation) => ...) — cambia la etiqueta según si estamos creando o editando

$operation es inyectado por Filament y vale 'create' o 'edit'.

unique(ignoreRecord: true)

El campo usuario debe ser único, pero al editar un usuario existente, ignoreRecord: true excluye el registro actual de la validación. Sin esto, editar un usuario fallaría porque el mismo nombre de usuario "ya existe".

4 Roles filtrados por jerarquía

UserForm.php — roles y método privado
Select::make('roles')
    ->relationship('roles', 'name')
    ->searchable()->preload()
    ->options(fn () => self::getRolesDisponibles())
    ->required()->label('Rol'),

Select::make('centro_salud_id')
    ->relationship('centroSalud', 'nombre')
    ->searchable()->preload()
    ->required(fn () => !auth()->user()?->hasRole('superadmin'))
    ->disabled(fn () => auth()->user()?->hasRole('admin'))
    ->default(fn () => auth()->user()?->centro_salud_id),

// ─── Método privado ───

private static function getRolesDisponibles(): array
{
    $user = auth()->user();
    if ($user?->hasRole('superadmin')) {
        return [
            'superadmin'  => 'Superadmin',
            'admin'       => 'Admin (Médico)',
            'registrador' => 'Registrador (Enfermero/a)',
        ];
    }
    return [
        'registrador' => 'Registrador (Enfermero/a)',
    ];
}
Jerarquía de roles — escalamiento de privilegios prevenido

getRolesDisponibles() previene escalamiento de privilegios:

  • Superadmin: puede asignar los 3 roles (superadmin, admin, registrador)
  • Admin: solo puede crear registradores — no puede crear otros admins ni superadmins

Esto es una defensa en profundidad. Incluso si alguien manipula el formulario, el backend solo permite las opciones declaradas.

centro_salud_id con required condicional

El superadmin puede no tener centro asignado (es global), por eso required(fn () => !hasRole('superadmin')) lo hace opcional solo para superadmin. Admin no puede cambiar el centro (disabled + default).

5 UsersTable — acción "Aprobar"

app/Filament/Resources/Users/Tables/UsersTable.php
->columns([
    TextColumn::make('name')->sortable()->searchable(),
    TextColumn::make('apellidos')->sortable()->searchable(),
    TextColumn::make('usuario')->searchable(),
    TextColumn::make('roles.name')->badge(),
    TextColumn::make('centroSalud.nombre')->searchable(),
    IconColumn::make('activo')->boolean(),
    TextColumn::make('ultimo_acceso')
        ->dateTime('d/m/Y H:i')->sortable()
        ->toggleable(isToggledHiddenByDefault: true),
])
->filters([
    SelectFilter::make('activo')
        ->options([1 => 'Activos', 0 => 'Inactivos']),
])
->defaultSort('activo', 'asc') // pendientes primero
->recordActions([
    Action::make('aprobar')
        ->label('Aprobar')
        ->icon('heroicon-o-check-circle')
        ->color('success')
        ->visible(fn ($record) => !$record->activo)
        ->requiresConfirmation()
        ->modalHeading('Aprobar usuario')
        ->modalDescription(fn ($record) =>
            "¿Activar la cuenta de {$record->name}?")
        ->action(fn ($record) =>
            $record->update(['activo' => true])
        ),
    EditAction::make(),
])
defaultSort('activo', 'asc') — pendientes primero

Los usuarios inactivos (activo = false = 0) aparecen primero. Combinado con la acción "Aprobar" que solo se muestra en inactivos, el flujo de aprobación es intuitivo: los pendientes están arriba con un botón verde para activar.

roles.name como badge

TextColumn::make('roles.name') accede a la relación many-to-many de Spatie Permission. ->badge() muestra el nombre del rol como un chip de color. Si un usuario tiene múltiples roles, se muestran todos como badges separados.

6 Resumen

Lo que aprendimos

  • whereHas('roles') — query scoping por relación polimórfica
  • Password condicional — required/dehydrated/label dinámicos con $operation
  • unique(ignoreRecord: true) — exclusión del registro actual en validación unique
  • getRolesDisponibles() — opciones de Select filtradas por jerarquía de roles
  • required() condicional — campo obligatorio según el rol del creador
  • Acción "Aprobar" — visible() + action() para aprobación rápida
  • defaultSort('activo', 'asc') — pendientes siempre arriba
