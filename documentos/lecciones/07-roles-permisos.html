1 Autenticación vs Autorización

Analogía

La autenticación es tu carnet de identidad (prueba quién eres). La autorización es tu credencial de trabajo (prueba qué puedes hacer). Puedes tener carnet pero no tener credencial para entrar a la sala de cirugía.

ConceptoPreguntaEn SICSAL

Autorización (esta lección)¿Qué puedes hacer?Roles: superadmin, admin, registrador

2 Spatie Permission: el paquete

spatie/laravel-permission es el paquete más popular de Laravel para manejar roles y permisos. Crea 5 tablas en la base de datos:

roles
superadmin, admin, registrador
←→
model_has_roles
user_id ↔ role_id
←→
users
cada usuario del sistema
SICSAL usa solo roles (sin permisos individuales). La autorización se basa en el rol del usuario.
¿Roles vs Permisos?

Los permisos son acciones específicas (editar-persona, ver-informe). Los roles agrupan permisos. SICSAL simplifica: solo usa roles, porque las reglas del negocio son claras (un registrador siempre puede lo mismo en cualquier centro).

3 Los 3 roles de SICSAL

superadmin

Ve todo. Accede a todos los centros de salud.

  • Crear/editar centros de salud
  • Crear/aprobar usuarios
  • Ver reportes de todos los centros
  • Importar datos .ves
  • centro_salud_id = null
admin

Solo su centro. Médico responsable.

  • Gestionar padrón de personas
  • Registrar prestaciones
  • Cerrar/reabrir meses
  • Ver reportes de SU centro
  • centro_salud_id = N
registrador

Solo su centro. Personal de estadística.

  • Registrar datos clínicos
  • Registrar prestaciones mensuales
  • Ver reportes de SU centro
  • No puede cerrar meses
  • centro_salud_id = N

4 HasRoles en el modelo User

Para que un modelo tenga roles, solo necesita usar el trait HasRoles:

app/Models/User.php — Partes clave
use Spatie\Permission\Traits\HasRoles;

class User extends Authenticatable implements FilamentUser
{
    use HasFactory, HasRoles, Notifiable, TwoFactorAuthenticatable;
                     // ↑ Este trait agrega todos los métodos de roles

    protected $fillable = [
        'name', 'apellidos', 'usuario', 'email',
        'password', 'centro_salud_id', 'activo', 'ultimo_acceso',
    ];

    protected $hidden = [
        'password', 'two_factor_secret',
        'two_factor_recovery_codes', 'remember_token',
    ];

    protected function casts(): array
    {
        return [
            'password' => 'hashed',  // Auto-hash al guardar
            'activo'   => 'boolean',
        ];
    }

    // Relación: un usuario pertenece a un centro de salud
    public function centroSalud(): BelongsTo
    {
        return $this->belongsTo(CentroSalud::class);
    }
}

Con HasRoles, el modelo User obtiene estos métodos:

Métodos disponibles gracias a HasRoles
// Asignar un rol (se usa en UserSeeder)
$user->assignRole('admin');

// Verificar si tiene un rol específico
$user->hasRole('superadmin');         // true o false

// Verificar si tiene alguno de varios roles
$user->hasAnyRole(['superadmin', 'admin']);  // true si tiene al menos uno

// Quitar un rol
$user->removeRole('registrador');

// Obtener todos los roles del usuario
$user->getRoleNames();             // ['admin']

5 canAccessPanel: la puerta de Filament

Filament requiere que el modelo User implemente FilamentUser con el método canAccessPanel(). Este método decide quién puede entrar al panel de administración:

app/Models/User.php — canAccessPanel()
use Filament\Models\Contracts\FilamentUser;

class User extends Authenticatable implements FilamentUser
{
    public function canAccessPanel(Panel $panel): bool
    {
        return $this->activo                              // 1. Cuenta aprobada
            && $this->hasAnyRole([                         // 2. Tiene algún rol
                   'superadmin',
                   'admin',
                   'registrador',
               ]);
    }
}
Doble candado

Para acceder al panel Filament se necesitan dos condiciones: activo = true (aprobado por admin) Y tener un rol asignado. Sin ambas, te rechaza con un 403 Forbidden.

6 Filtrar datos por centro de salud

La regla más importante de SICSAL: un admin/registrador solo ve datos de SU centro. Esto se implementa en el getEloquentQuery() de cada Resource:

app/Filament/Resources/Personas/PersonaResource.php
public static function getEloquentQuery(): Builder
{
    $query = parent::getEloquentQuery();
    $user  = auth()->user();

    // Si NO es superadmin → filtrar por su centro
    if ($user && !$user->hasRole('superadmin')) {
        $query->where('centro_salud_id', $user->centro_salud_id);
    }

    return $query;
}

7 HasCentroSelector: el trait compartido

Para las páginas Livewire (informes, prestaciones), se usa un trait reutilizable que aplica la misma lógica:

app/Concerns/HasCentroSelector.php
trait HasCentroSelector
{
    // El ID del centro actual viene de la URL (?centroSaludId=5)
    #[Url]
    public int $centroSaludId = 0;

    public function mountHasCentroSelector(): void
    {
        $user = auth()->user();

        // Si no viene de la URL, usar el centro del usuario
        if ($this->centroSaludId === 0) {
            $this->centroSaludId = $user?->centro_salud_id ?? 1;
        }

        // Seguridad: forzar centro para no-superadmins
        if ($user && !$user->hasRole('superadmin') && $user->centro_salud_id) {
            $this->centroSaludId = (int) $user->centro_salud_id;
        }
    }

    // Evitar cambio de centro via URL para no-superadmins
    public function updatingCentroSaludId(int $value): void
    {
        $user = auth()->user();
        if ($user && !$user->hasRole('superadmin') && $user->centro_salud_id) {
            $this->centroSaludId = (int) $user->centro_salud_id;
        }
    }

    // Solo superadmin ve el selector con todos los centros
    public function getCentrosDisponiblesProperty(): array
    {
        $user = auth()->user();
        if (!$user || !$user->hasRole('superadmin')) {
            return [];  // No muestra selector
        }

        return CentroSalud::where('activo', true)
            ->orderBy('nombre')->pluck('nombre', 'id')->toArray();
    }
}
Seguridad en el selector

El método updatingCentroSaludId() es un guardia: si un usuario no-superadmin intenta cambiar el centro de salud manipulando la URL (?centroSaludId=99), el sistema fuerza su centro asignado. No puede espiar datos de otros centros.

8 Ejemplos reales de verificación de rol

Estos patrones se repiten a lo largo de todo el código de SICSAL:

Patrón 1: Solo admin/superadmin pueden cerrar meses
public function cerrarMes(): void
{
    $user = auth()->user();

    // Solo admin y superadmin pueden cerrar meses
    if (!$user->hasAnyRole(['superadmin', 'admin'])) {
        return;  // Registrador no puede
    }

    MesCerrado::updateOrCreate(
        ['centro_salud_id' => $user->centro_salud_id, ...],
        ['cerrado_por' => $user->id]
    );
}
Patrón 2: Superadmin ve todo, otros ven su centro
// En cualquier query Eloquent:
$query = Persona::query();

if (!auth()->user()->hasRole('superadmin')) {
    $query->where('centro_salud_id', auth()->user()->centro_salud_id);
}
Patrón 3: Mostrar/ocultar UI según rol (Blade)
{{-- En una vista Blade --}}
@role('superadmin')
    <!-- Solo superadmin ve esto -->
    <select>...selector de centros...</select>
@endrole

@hasanyrole('superadmin|admin')
    <!-- Admin y superadmin pueden cerrar el mes -->
    <button>Cerrar Mes</button>
@endhasanyrole

Resumen de la Lección 07

  • Spatie Permission maneja roles con el trait HasRoles en el modelo User.
  • SICSAL tiene 3 roles: superadmin (ve todo), admin (su centro), registrador (su centro, más limitado).
  • canAccessPanel() exige activo + rol para entrar a Filament.
  • Cada Resource filtra datos con getEloquentQuery() según el centro_salud_id del usuario.
  • HasCentroSelector es un trait reutilizable que impide manipulación de URL por no-superadmins.
  • Métodos clave: hasRole(), hasAnyRole(), assignRole().
