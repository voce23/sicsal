1 ¿Qué es una migración?

Una migración es un archivo PHP que describe cómo crear o modificar una tabla en la base de datos. En vez de abrir MySQL y escribir SQL a mano, escribes código PHP que Laravel traduce a SQL automáticamente.

❌ Sin migraciones (la forma vieja)

Abrir phpMyAdmin, crear tablas con clics, olvidar qué cambios hiciste, imposible replicar en otro servidor.

✅ Con migraciones (la forma moderna)

Escribir un archivo PHP con la estructura. Se puede ejecutar en cualquier servidor con un solo comando. Todo queda versionado en Git.

Analogía

Las migraciones son como un plano de arquitectura. En vez de construir la casa directamente, primero dibujas el plano. Después puedes construir la misma casa cuantas veces quieras siguiendo el plano.

2 Crear tu primera migración

Para crear una migración, usas el comando artisan:

Terminal
$ php artisan make:migration create_municipios_table

   INFO  Migration [database/migrations/2026_04_03_015720_create_municipios_table.php] created successfully.
El nombre del archivo

Laravel agrega automáticamente una marca de tiempo al nombre: 2026_04_03_015720_create_municipios_table.php. Esto asegura que las migraciones se ejecuten en el orden correcto (las tablas que dependen de otras se crean después).

El archivo generado tiene esta estructura básica:

database/migrations/xxxx_create_municipios_table.php
<?php

use Illuminate\Database\Migrations\Migration;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Support\Facades\Schema;

return new class extends Migration
{
    /**
     * up() → Lo que HACE la migración (crear tabla, agregar columna...)
     */
    public function up(): void
    {
        Schema::create('municipios', function (Blueprint $table) {
            $table->id();                              // Columna id autoincremental
            // ← Aquí defines tus columnas
            $table->timestamps();                      // created_at y updated_at
        });
    }

    /**
     * down() → Lo que DESHACE la migración (borrar tabla...)
     */
    public function down(): void
    {
        Schema::dropIfExists('municipios');
    }
};
up() y down()
  • up() — Se ejecuta con php artisan migrate. Crea la tabla.
  • down() — Se ejecuta con php artisan migrate:rollback. Deshace los cambios (borra la tabla).

3 Tipos de columna más usados

La variable $table (de tipo Blueprint) te permite definir columnas. Estos son los tipos que SICSAL usa:

Modificadores de columna

Modificadores comunes
$table->string('ci', 20)->nullable();          // Puede ser NULL
$table->boolean('activo')->default(true);        // Valor por defecto
$table->integer('poblacion_ine')->default(0);    // Número con valor por defecto
$table->string('codigo')->unique();             // No se puede repetir
$table->integer('edad')->unsigned();             // Solo positivos

4 Foreign Keys (claves foráneas)

Una clave foránea (foreign key) conecta una tabla con otra. Por ejemplo, cada persona pertenece a un centro de salud. Para expresar esto, la tabla personas tiene una columna centro_salud_id que apunta a la tabla centros_salud.

RELACIÓN: persona → centro_salud
personas
centro_salud_id →
centros_salud.id
Dos formas de crear foreign keys
// Forma corta (recomendada en Laravel 13)
$table->foreignId('centro_salud_id')->constrained('centros_salud');
// Crea: BIGINT UNSIGNED + FOREIGN KEY → centros_salud(id)

// Forma corta con nullable (la relación es opcional)
$table->foreignId('created_by')->nullable()->constrained('users');
// Puede ser NULL (no siempre sabemos quién creó el registro)
Orden de las migraciones

La tabla referenciada debe existir antes que la tabla que la referencia. Por eso los nombres de archivo tienen marca de tiempo: 015721_centros_salud se crea antes que 015726_personas.

5 Ejemplo real: tabla centros_salud

Veamos cómo SICSAL define la tabla de centros de salud. Es una tabla sencilla con pocos campos:

database/migrations/2026_04_03_015721_create_centros_salud_table.php
return new class extends Migration
{
    public function up(): void
    {
        Schema::create('centros_salud', function (Blueprint $table) {
            $table->id();                                           // PK autoincremental
            $table->foreignId('municipio_id')                        // FK → municipios
                  ->constrained('municipios');
            $table->string('nombre', 150);                          // Nombre del centro
            $table->string('codigo_snis', 20)->nullable();          // Código del SNIS
            $table->enum('subsector',                                // Tipo de sector
                ['Público', 'Seguro Social',
                 'Privado', 'ONG'])->default('Público');
            $table->string('red_salud', 100)->nullable();         // Red a la que pertenece
            $table->integer('poblacion_ine')->default(0);         // Población INE total
            $table->boolean('activo')->default(true);             // Borrado lógico
            $table->timestamps();                                   // created_at, updated_at
        });
    }

    public function down(): void
    {
        Schema::dropIfExists('centros_salud');
    }
};

Esta tabla genera en MySQL:

centros_salud
id PK | municipio_id FK | nombre VARCHAR(150) | codigo_snis VARCHAR(20) NULL | subsector ENUM | red_salud VARCHAR(100) NULL | poblacion_ine INT DEFAULT 0 | activo BOOL | created_at updated_at

6 Ejemplo real: tabla personas

La tabla personas es más compleja. Tiene múltiples foreign keys, enums y campos opcionales:

database/migrations/2026_04_03_015726_create_personas_table.php
Schema::create('personas', function (Blueprint $table) {
    $table->id();

    // ─── Relaciones ───
    $table->foreignId('centro_salud_id')->constrained('centros_salud');  // A qué centro pertenece
    $table->foreignId('comunidad_id')->constrained('comunidades');      // De qué comunidad es

    // ─── Datos personales ───
    $table->string('nombres', 100);
    $table->string('apellidos', 100);
    $table->date('fecha_nacimiento');
    $table->enum('sexo', ['M', 'F']);
    $table->string('ci', 20)->nullable();              // CI es optional (menores)

    // ─── Seguro y estado migratorio ───
    $table->enum('tipo_seguro',
        ['SUS', 'privado', 'ninguno'])->default('ninguno');
    $table->enum('estado',
        ['residente', 'temporal', 'migrado'])->default('residente');
    $table->string('destino_migracion', 150)->nullable();
    $table->date('fecha_migracion')->nullable();

    // ─── Sistema ───
    $table->string('grupo_etareo', 30)->nullable();
    $table->date('fecha_registro');
    $table->text('observaciones')->nullable();
    $table->boolean('activo')->default(true);              // Borrado lógico
    $table->foreignId('created_by')->nullable()->constrained('users');
    $table->timestamps();
});
Convención: activo = true (borrado lógico)

En SICSAL nunca borramos registros de la base de datos. En vez de eliminar, ponemos activo = false. Esto se llama "borrado lógico" y nos permite recuperar datos si fue un error. Es una regla del proyecto.

7 Ejemplo avanzado: tabla vacunas_ninos

Esta tabla demuestra cómo usar enums extensos para limitar las opciones válidas directamente en la base de datos:

database/migrations/2026_04_04_010000_create_vacunas_ninos_table.php
Schema::create('vacunas_ninos', function (Blueprint $table) {
    $table->id();
    $table->foreignId('persona_id')->constrained('personas');

    $table->enum('tipo_vacuna', [
        'BCG', 'HepB_neonatal',                        // Neonato
        'Pentavalente_1', 'Pentavalente_2',             // Pentavalente
        'Pentavalente_3', 'Pentavalente_4_refuerzo',
        'Pentavalente_5_refuerzo',
        'IPV_1', 'bOPV_2', 'IPV_3',                 // Polio
        'bOPV_4', 'bOPV_5',
        'Antirotavirica_1', 'Antirotavirica_2',       // Rotavirus
        'Antineumococica_1', 'Antineumococica_2',     // Neumococo
        'Antineumococica_3',
        'Influenza_1', 'Influenza_2',                 // Influenza
        'Influenza_unica',
        'SRP_1', 'SRP_2', 'Antiamarilica',          // SRP + Fiebre amarilla
        'VPH_1', 'VPH_2',                               // VPH
        'dT_1', 'dT_2', 'dT_3', 'dT_4', 'dT_5',   // Toxoide
        'SR',
    ]);

    $table->date('fecha_aplicacion');
    $table->enum('dentro_fuera', ['dentro', 'fuera'])
          ->default('dentro');                               // Dentro/fuera del centro
    $table->string('lote', 50)->nullable();                  // Nº lote de la vacuna
    $table->foreignId('aplicado_por')->nullable()
          ->constrained('users');                              // Quién la aplicó
    $table->tinyInteger('mes');                              // Mes (1-12)
    $table->year('anio');                                     // Año
    $table->timestamps();
});
¿Por qué un enum tan largo?

El esquema PAI (Programa Ampliado de Inmunización) de Bolivia define vacunas específicas con dosis numeradas. Cada dosis es un registro separado. El enum evita errores de escritura: MySQL rechazará cualquier valor que no esté en la lista.

8 Modificar una tabla existente

¿Qué pasa si necesitas agregar una columna después de crear la tabla? Creas una nueva migración que modifica la tabla existente:

Terminal
$ php artisan make:migration add_coordenadas_to_centros_salud
database/migrations/2026_04_07_203512_add_coordenadas_to_centros_salud.php
return new class extends Migration
{
    public function up(): void
    {
        Schema::table('centros_salud', function (Blueprint $table) {
            // Schema::table() modifica una tabla existente
            // (a diferencia de Schema::create() que crea una nueva)
            $table->decimal('latitud', 10, 7)->nullable();
            $table->decimal('longitud', 10, 7)->nullable();
        });
    }

    public function down(): void
    {
        Schema::table('centros_salud', function (Blueprint $table) {
            $table->dropColumn(['latitud', 'longitud']);
        });
    }
};
Regla importante

NUNCA modifiques una migración que ya se ejecutó en producción. Siempre crea una migración nueva. Las migraciones son como un historial — cada cambio es un nuevo archivo.

9 Comandos artisan para migraciones

Terminal — Comandos de migraciones
# Ejecutar todas las migraciones pendientes
$ php artisan migrate

# Ver el estado de las migraciones
$ php artisan migrate:status

# Deshacer la última migración
$ php artisan migrate:rollback

# Borrar todo y recrear (solo desarrollo, NUNCA en producción)
$ php artisan migrate:fresh

# Borrar y recrear + ejecutar seeders
$ php artisan migrate:fresh --seed

# Crear una nueva migración
$ php artisan make:migration create_nombre_tabla_table
$ php artisan make:migration add_columna_to_tabla
Cuidado con migrate:fresh

migrate:fresh borra TODAS las tablas y las recrea. Solo úsalo en desarrollo. En producción, siempre usa migrate (que solo ejecuta migraciones nuevas).

Sistema dual de datos

SICSAL tiene dos sistemas de datos paralelos: tablas individuales (un registro por persona/evento) y tablas agregadas prest_* (totales mensuales por centro). Los Observers (Lección 14) sincronizan automáticamente de uno al otro.

Resumen de la Lección 03

  • Las migraciones son archivos PHP que crean/modifican tablas — como planos de construcción.
  • Se crean con php artisan make:migration y se ejecutan con php artisan migrate.
  • Cada migración tiene up() (aplicar) y down() (deshacer).
  • Tipos comunes: string, integer, date, boolean, enum, foreignId.
  • Las foreign keys conectan tablas entre sí con foreignId()->constrained().
  • Para modificar una tabla existente, crea una nueva migración con Schema::table().
  • SICSAL usa borrado lógico (activo = false) en vez de eliminar registros.
  • El proyecto tiene 51 migraciones que crean un sistema dual: datos individuales + datos agregados SNIS.
