1 ¿Qué es la autenticación?

Autenticación es el proceso de verificar quién eres. Es diferente de autorización (qué puedes hacer, que veremos en la Lección 07).

Autenticación

"¿Eres quien dices ser?"

→ Login con email + contraseña

→ Verificación de email

→ Autenticación en dos pasos (2FA)

Autorización

"¿Tienes permiso para esto?"

→ Roles: superadmin, admin, registrador

→ Políticas: ¿puedes editar esta persona?

→ Lección 07

2 Laravel Fortify: autenticación sin vistas

Fortify es un paquete de Laravel que proporciona toda la lógica de autenticación (login, registro, reset de contraseña, 2FA) pero sin ninguna vista prediseñada. Tú decides cómo se ven las pantallas.

¿Por qué Fortify y no Breeze o Jetstream?

SICSAL usa Filament para el panel de administración. Fortify da control total sobre las vistas y la lógica sin imponer un diseño. Breeze y Jetstream incluyen vistas propias que entrarían en conflicto con Filament.

Blade Views
login, register, reset...
→
Fortify Routes
/login, /register, /forgot-password
→
Fortify Actions
CreateNewUser, ResetUserPassword
Fortify se encarga de las rutas y la lógica. Tú creas las vistas y las Actions.

3 Configuración: config/fortify.php

config/fortify.php — Partes clave
return [
    'guard'     => 'web',              // Guard de autenticación
    'username'  => 'email',             // Campo para login (email, no usuario)
    'home'      => '/dashboard',         // A dónde ir después del login

    'lowercase_usernames' => true,         // Email siempre en minúscula

    'limiters' => [
        'login'      => 'login',           // Máx 5 intentos/minuto
        'two-factor' => 'two-factor',      // Máx 5 intentos/minuto
    ],

    'features' => [
        Features::registration(),          // ✅ Registro público habilitado
        Features::resetPasswords(),        // ✅ "Olvidé mi contraseña"
        Features::emailVerification(),     // ✅ Verificación de email
        Features::twoFactorAuthentication([  // ✅ 2FA con Google Auth
            'confirm'         => true,
            'confirmPassword' => true,
        ]),
    ],
];
Registro público ≠ acceso libre

Aunque el registro está habilitado, los usuarios nuevos se crean con activo = false. No pueden hacer nada hasta que un admin los apruebe. Esto es intencional: el registro es una solicitud de acceso, no un acceso inmediato.

4 FortifyServiceProvider: el cerebro

Este proveedor personaliza Fortify para SICSAL. Tiene tres partes:

4.1 — Respuesta personalizada al Login

app/Providers/FortifyServiceProvider.php — register()
public function register(): void
{
    // Personalizar qué pasa después del login
    $this->app->singleton(LoginResponse::class, function () {
        return new class implements LoginResponse {
            public function toResponse($request)
            {
                // Si el usuario está activo → dashboard
                // Si NO está activo → página "pendiente"
                $redirect = $request->user()->activo
                    ? config('fortify.home')   // /dashboard
                    : route('pendiente');      // /pendiente

                return redirect()->intended($redirect);
            }
        };
    });

    // Tras registro → siempre a /pendiente
    $this->app->singleton(RegisterResponse::class, function () {
        return new class implements RegisterResponse {
            public function toResponse($request)
            {
                return redirect()->route('pendiente');
            }
        };
    });
}

4.2 — Vistas personalizadas

app/Providers/FortifyServiceProvider.php — configureViews()
private function configureViews(): void
{
    Fortify::loginView(fn () =>
        view('pages::auth.login'));

    Fortify::registerView(fn () =>
        view('pages::auth.register'));

    Fortify::resetPasswordView(fn () =>
        view('pages::auth.reset-password'));

    Fortify::requestPasswordResetLinkView(fn () =>
        view('pages::auth.forgot-password'));

    Fortify::verifyEmailView(fn () =>
        view('pages::auth.verify-email'));

    Fortify::twoFactorChallengeView(fn () =>
        view('pages::auth.two-factor-challenge'));

    Fortify::confirmPasswordView(fn () =>
        view('pages::auth.confirm-password'));
}

4.3 — Rate Limiting

app/Providers/FortifyServiceProvider.php — configureRateLimiting()
private function configureRateLimiting(): void
{
    // Máximo 5 intentos de login por minuto por email+IP
    RateLimiter::for('login', function (Request $request) {
        $key = Str::lower(
            $request->input(Fortify::username())
        ) . '|' . $request->ip();

        return Limit::perMinute(5)->by($key);
    });
}
¿Qué es Rate Limiting?

Limita cuántas veces alguien puede intentar una acción en un período de tiempo. Si alguien intenta adivinar una contraseña, después de 5 intentos en 1 minuto, se bloquea temporalmente. Protege contra ataques de fuerza bruta.

5 Registro: CreateNewUser (activo = false)

Cuando alguien llena el formulario de registro, Fortify llama a esta clase:

app/Actions/Fortify/CreateNewUser.php
class CreateNewUser implements CreatesNewUsers
{
    use PasswordValidationRules;

    public function create(array $input): User
    {
        // 1. Validar los datos
        Validator::make($input, [
            'name'            => ['required', 'string', 'max:100'],
            'apellidos'       => ['required', 'string', 'max:100'],
            'email'           => ['required', 'email', 'max:255',
                                    Rule::unique(User::class)],
            'centro_salud_id' => ['required',
                                    Rule::exists(CentroSalud::class, 'id')],
            'password'        => $this->passwordRules(),
        ])->validate();

        // 2. Crear usuario con activo = false
        return User::create([
            'name'            => $input['name'],
            'apellidos'       => $input['apellidos'],
            'email'           => $input['email'],
            'centro_salud_id' => $input['centro_salud_id'],
            'password'        => $input['password'],
            'activo'          => false,  // ← CLAVE: inactivo hasta aprobación
        ]);
    }
}
Regla de negocio: activo = false

Todo usuario nuevo se crea con activo = false. No tiene rol asignado. Solo cuando un admin o superadmin lo apruebe desde el panel Filament, se activa la cuenta y se le asigna un rol. Esto impide el acceso no autorizado.

6 Middleware RequireAprobado

Un middleware es un filtro que se ejecuta antes (o después) de cada petición HTTP. Es como un guardia en la puerta:

app/Http/Middleware/RequireAprobado.php
class RequireAprobado
{
    public function handle(Request $request, Closure $next): Response
    {
        // Si hay usuario logueado Y no está activo
        if ($request->user() && !$request->user()->activo) {
            return redirect()->route('pendiente');
        }

        return $next($request);  // Puede pasar
    }
}

El middleware se registra en bootstrap/app.php con un alias:

bootstrap/app.php
return Application::configure(basePath: dirname(__DIR__))
    ->withRouting(
        web: __DIR__.'/../routes/web.php',
        commands: __DIR__.'/../routes/console.php',
        health: '/up',
    )
    ->withMiddleware(function (Middleware $middleware): void {
        $middleware->alias([
            'aprobado' => \App\Http\Middleware\RequireAprobado::class,
        ]);
    });

Ahora, en las rutas, podemos proteger páginas así:

Uso en rutas
// Solo usuarios logueados Y aprobados pueden ver el dashboard
Route::middleware(['auth', 'aprobado'])->group(function () {
    Route::get('/dashboard', function () {
        return view('dashboard');
    });
});

8 Features de seguridad

Feature¿Qué hace?¿Cómo protege?
registration() Formulario de auto-registro activo=false hasta aprobación del admin
resetPasswords() "Olvidé mi contraseña" por email Token temporal, expira en 60 min
emailVerification() Confirmar email real Evita registros con emails falsos
twoFactorAuthentication() Código de 6 dígitos (Google Authenticator) Incluso si roban la contraseña, necesitan el teléfono
Rate Limiting Máx 5 intentos/minuto Bloquea ataques de fuerza bruta
bcrypt Hash irreversible de contraseñas Contraseñas nunca se guardan en texto plano
CSRF Protection

Cada formulario incluye @csrf — un token único que Laravel verifica. Esto impide que sitios externos envíen formularios falsos a tu aplicación (ataque Cross-Site Request Forgery).

Resumen de la Lección 06

  • Fortify proporciona lógica de autenticación sin imponer vistas.
  • Archivos clave: config/fortify.php (configuración), FortifyServiceProvider (personalización), Actions/ (lógica).
  • Usuarios nuevos se crean con activo = false — requieren aprobación de un admin.
  • El middleware RequireAprobado bloquea el acceso a usuarios no aprobados en cada petición.
  • Rate limiting, 2FA, verificación de email y bcrypt forman las capas de seguridad.
  • Las vistas de autenticación están en resources/views/pages/auth/.
