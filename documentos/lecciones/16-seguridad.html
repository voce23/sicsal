1 Secretos en .env (nunca en código)

.env — Variables sensibles
# Base de datos
DB_PASSWORD=mi_contraseña_segura

# Passwords de seeders (para desarrollo)
SUPERADMIN_PASSWORD=cambiar_en_produccion
ADMIN_PASSWORD=cambiar_en_produccion

# Clave de la aplicación (generada por Laravel)
APP_KEY=base64:xxxxxxxxxxxxxxxxxxxxxxxxxxx
UserSeeder.php — Usar env() en vez de hardcoded
// ❌ MAL: contraseña visible en el código fuente
'password' => Hash::make('admin123'),

// ✅ BIEN: leer de variable de entorno
'password' => Hash::make(env('ADMIN_PASSWORD', 'changeme')),
Regla de oro

El archivo .env nunca se sube al repositorio (está en .gitignore). Cada servidor tiene su propio .env con sus propias credenciales. Si alguien ve tu código en GitHub, no podrá ver tus contraseñas.

2 Protección CSRF

Blade — Token CSRF en formularios
<!-- Laravel incluye esto automáticamente en formularios Blade -->
<form method="POST" action="/ruta">
    @csrf   <!-- Genera un token oculto único por sesión -->
    ...
</form>

<!-- Se convierte en: -->
<input type="hidden" name="_token" value="abc123xyz..." />
¿Qué es CSRF?

Cross-Site Request Forgery: un atacante engaña a un usuario autenticado para que envíe un formulario malicioso. El token @csrf de Laravel verifica que el formulario fue generado por tu propia aplicación. Livewire lo maneja automáticamente.

3 Middleware RequireAprobado

app/Http/Middleware/RequireAprobado.php
class RequireAprobado
{
    public function handle(Request $request, Closure $next)
    {
        // Si el usuario no está activo → redirigir a /pendiente
        if (auth()->check() && !auth()->user()->activo) {
            return redirect('/pendiente');
        }
        return $next($request);
    }
}
bootstrap/app.php — Registrar el middleware
return Application::configure(...)
    ->withMiddleware(function (Middleware $middleware) {
        $middleware->alias([
            'aprobado' => RequireAprobado::class,
        ]);
    });
Usuario se registra → activo = false
↓
Intenta acceder al panel → middleware 'aprobado'
↓
¿activo == false? → Redirige a /pendiente
↓
Admin activa al usuario → activo = true
↓
Ahora puede acceder al panel normalmente

4 Autorización por centro de salud

app/Concerns/HasCentroSelector.php
trait HasCentroSelector
{
    public $centroSaludId;

    // Al cambiar centro, verificar que el usuario tenga permiso
    public function updatingCentroSaludId($value)
    {
        $user = auth()->user();

        // Superadmin puede ver cualquier centro
        if ($user->hasRole('superadmin')) return;

        // Admin/Registrador solo pueden ver SU centro
        if ($value != $user->centro_salud_id) {
            abort(403, 'No autorizado para este centro.');
        }
    }
}
Defensa contra manipulación

Sin este guard, un usuario podría cambiar el centro de salud en la URL o con DevTools y ver datos de otro centro. El updatingCentroSaludId se ejecuta automáticamente cada vez que Livewire intenta cambiar esa propiedad.

5 Validación de entrada

Filament — Validación automática en formularios
// Los campos de Filament validan automáticamente
TextInput::make('nombre')
    ->required()               // No puede estar vacío
    ->maxLength(255)           // Máximo 255 caracteres
    ->rules(['alpha_spaces']) // Solo letras y espacios

Select::make('centro_salud_id')
    ->required()
    ->relationship('centroSalud', 'nombre') // Verifica que exista en BD
Livewire — Validación con atributos PHP 8
// En Comentarios.php
#[Validate('required|max:500')]
public string $contenido = '';

public function guardar()
{
    $this->validate();  // Valida todas las propiedades con #[Validate]

    // Sanitizar HTML para prevenir XSS
    $texto = strip_tags($this->contenido);
}
strip_tags() contra XSS

XSS (Cross-Site Scripting) es cuando un atacante inyecta JavaScript malicioso en tu página. strip_tags() elimina todas las etiquetas HTML del texto del usuario, previniendo que inyecte <script>.

6 Restricciones en base de datos

Migration — Restricciones UNIQUE
// Evitar duplicados a nivel de base de datos
Schema::table('personas', function (Blueprint $table) {
    // Una persona solo puede tener un CI en un centro
    $table->unique(
        ['ci', 'centro_salud_id'],
        'personas_ci_centro_unique'
    );
});

Schema::table('centros_salud', function (Blueprint $table) {
    // Cada centro tiene un código SNIS único
    $table->unique('codigo_snis');
});
Migration — Índices de rendimiento
// Acelerar consultas frecuentes
Schema::table('personas', function (Blueprint $table) {
    $table->index('estado');  // Filtrar por activo/migrado/fallecido
});

Schema::table('centros_salud', function (Blueprint $table) {
    $table->index('codigo_snis'); // Buscar centro por código
});
¿Por qué restricciones en BD?

La validación en PHP atrapa el 99% de los errores. Pero si hay un bug, una condición de carrera, o alguien accede directamente a la BD, las restricciones UNIQUE e índices son la última línea de defensa.

Resumen de la Lección 16

  • Secretos: Siempre en .env, nunca hardcoded. .env está en .gitignore.
  • CSRF: @csrf en formularios Blade. Livewire lo maneja automáticamente.
  • Middleware: RequireAprobado bloquea usuarios no activados.
  • Autorización: HasCentroSelector usa abort(403) si intentan cambiar de centro.
  • Validación: Filament + #[Validate] + strip_tags() contra XSS.
  • Base de datos: UNIQUE constraints como última línea de defensa.
  • Shell: Whitelist de 56 tablas en VesMdbReader contra inyección de comandos.
  • La seguridad funciona en capas: si una falla, las demás siguen protegiendo.
